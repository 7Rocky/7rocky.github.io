    match(input('Selection:')):
        case '0':
            inp = input('To Add:')
            storage[id(inp)] = inp
            print(id(inp))
            del inp

            addr = input('To Remove:')
            if addr.isdecimal() and int(addr) in storage:

        case '2':
            addr = input('To Load:')
            if addr.isdecimal():

Source code analysis

The first thing that looks interesting in the script is the use of PyObj_FromPtr from _ctypes. This function takes an address and tries to parse a Python object. On the contrary, the use of id is also interesting. This function returns the address of a given Python object.

For instance, we have id(0) in the menu:

commands = f'''
Zero: {id(0)}
0) Add To Store
1) Remove From Store  
2) Load Address
'''.strip()

Allocation function

We can add a str object and the dict object will hold the address of the str as key and the str as value:

Python objects

After reading some resources on Python internals, we will see that the relevant structure is PyObject (aliased from _object, which is defined here):

/* Nothing is actually declared to be a PyObject, but every pointer to
 * a Python object can be cast to a PyObject*.  This is inheritance built
 * by hand.  Similarly every pointer to a variable-size Python object can,  
 * in addition, be cast to PyVarObject*.
 */
struct _object {
    _PyObject_HEAD_EXTRA
    Py_ssize_t ob_refcnt;
    PyTypeObject *ob_type;
};

Fake objects

While reading Exploiting a Use-After-Free for code execution in every version of Python 3, I understood that in order to exploit a Python application, one way is to craft fake objects in memory to confuse Python (similar to JavaScript exploits for browsers).

There are two similar structures that are mentioned in the article: bytearray and bytes.

>>> x = bytearray([1, 2, 3, 4, 5])

Fake object primitive

The previous function was not useful for exploitation, but we figured out how we can craft fake objects in memory and make Python load them. In order to get code execution, we will need to modify a function pointer. One approach is to modify a function pointer inside a PyTypeObject to overwrite a “magic/dunder” method (as shown in the article).

Here we must take into account that print(PyObj_FromPtr(int(addr))) is actually PyObj_FromPtr(int(addr)).__repr__(), or maybe it is first transformed to str with PyObj_FromPtr(int(addr)).__str__(). Therefore, we the idea is to enter a fake PyObject with a fake PyTypeObject that has a malicious function in the offset for __repr__ or __str__ (for instance, system).

Since we already bypassed PIE, we can call system with the Procedure Linkage Table (PLT), so that is not a problem (I used pwntools to get the exact address).
